runtime) that handle:\n- Webhook processing\n- Message intent detection\n- Database operations with RLS\n- Automatic scaling and reliability\n\n### Row Level Security (RLS)\nDatabase-level security ensuring:\n- Users can only see their own data\n- JWT from WhatsApp phone number enforces access\n- No data leakage between users\n- Applies at database level, not application level\n\n### Message Flow\n1. User sends WhatsApp message\n2. Meta forwards to webhook (`whatsapp-webhook` Edge Function)\n3. Bot detects intent using NLP\n4. Executes appropriate handler (attendance, timer, task, etc.)\n5. Queries database with RLS-enforced user scope\n6. Returns formatted response\n7. Logs interaction for analytics\n\n### Session Management\n- **In-Memory Cache**: Fast access within single request\n- **Database Persistence**: State survives across requests via `bot_sessions` table\n- **Used For**: Multi-step onboarding flow\n\n## Monitoring & Observability\n\n### Logs\n- **Edge Function Logs**: Supabase Dashboard → Functions → [Function] → Logs\n- **Message Logs**: Check `whatsapp_message_logs` table for delivery tracking\n- **Error Logs**: Console output in Edge Function logs\n\n### Metrics\n- **User Engagement**: Check `whatsapp_window_status` table\n- **Feature Usage**: Query message logs by type and intent\n- **System Health**: Monitor Edge Function error rates and latency\n\n### Alerts\n- Set up alerts for Edge Function errors in Supabase\n- Monitor WhatsApp Cloud API rate limits\n- Track daily active users via database queries\n\n## Common Tasks\n\n### Deploy a New Bot Feature\n1. Add feature handler to `processor.ts`\n2. Add NLP intent to `nlp.ts`\n3. Add message templates to `messages.ts`\n4. Test locally: `supabase functions serve`\n5. Deploy: `npx supabase functions deploy whatsapp-webhook`\n\n### Fix a Bug\n1. Check logs in Supabase Dashboard\n2. Review relevant handler in `processor.ts`\n3. Make fix and test locally\n4. Redeploy function\n5. Verify via test WhatsApp message\n\n### Add a Database Query\n1. Use `getUserClient(phone)` for RLS-enforced queries\n2. Use `supabaseAdmin` only for initial setup\n3. Always handle errors gracefully\n4. Return user-friendly error messages\n\n### Optimize Performance\n1. Add database indexes for frequently queried fields\n2. Cache static data (categories, templates) in memory\n3. Batch operations to reduce function invocations\n4. Monitor Edge Function execution times\n\n## Troubleshooting\n\n### Bot Not Responding\n- Check webhook URL in Meta Developer Portal\n- Verify `WA_VERIFY_TOKEN` matches in settings\n- Check Edge Function logs for errors\n- Verify WhatsApp token is valid\n\n### User Can't Mark Attendance\n- Verify user completed onboarding\n- Check subject exists and is active\n- Review RLS policies in database\n- Check user's `academics_enabled` flag\n\n### OTP Not Arriving\n- Verify `auth` Edge Function is deployed\n- Check WhatsApp token and phone ID\n- Verify phone number format (+country code required)\n- Check rate limiting (max 3 per 10 minutes)\n\n### Session State Not Persisting\n- Check `bot_sessions` table has records\n- Verify database connection in Edge Function\n- Look for errors in Edge Function logs\n- Ensure JWT_SECRET is set in environment\n\n## Further Reading\n\n- [Supabase Documentation](https://supabase.com/docs)\n- [Deno Manual](https://deno.land/manual)\n- [WhatsApp Cloud API Documentation](https://developers.facebook.com/docs/whatsapp/cloud-api/)\n- [JWT Authentication](https://jwt.io/)\n- [Row Level Security in PostgreSQL](https://www.postgresql.org/docs/current/ddl-rowsecurity.html)\n"
